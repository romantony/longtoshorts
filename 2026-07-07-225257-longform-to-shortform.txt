  What the pipeline is: a 15-stage Step Functions machine that takes one
  long-form video and produces N vertical shorts. Flow: probe the video вҶ’
  calculate segment cut points вҶ’ (optionally) run an ECS Fargate container for
  scene detection/subject tracking/diarization вҶ’ generate a shorts manifest вҶ’
  slice the global SRT into per-segment captions вҶ’ then four parallel Map stages
  (concurrency 4 each): split clips, convert to 9:16, upscale to 1080Г—1920, and
  finalize (burn captions, mix BGM, add title/end slides via ffmpeg). Status is
  pushed to Convex at 7 checkpoints, and assets flow through R2 (clips) and S3
  (analysis + manifests). Notably, the "GenerateShortHooks" stage is pure
  scaffolding вҖ” it's a Pass-through that sets hookVideoUrl: null; title slides
  are actually rendered by E2E-finalize-short.

  The doc covers the flow diagram, execution input schema, inferred input/output
  contracts for all 10 Lambdas + the ECS task, the three-tier error-handling
  model, storage layout, data-flow map, and an infrastructure checklist.

  Four real defects I found while analyzing (documented in В§8 вҖ” worth fixing
  before relying on the "resilient partial results" design):

  1. Failed Map items poison the next Map. A failed split emits {failed: true, 
  frameId, ...}, but the next Map's Lambda reads $.rawClipUrl from it вҖ” an
  unresolvable path raises States.Runtime, which bypasses all Catch blocks and
  kills the whole execution. The partial-failure design only works if downstream
  iterators skip failed: true items, and they don't.
  2. The failure handlers themselves can crash.
  SegmentSplitFailed/ConvertFailed/FinalizeFailed read $.frameId/$.frameNumber,
  but the item payloads built upstream only carry partNumber вҖ” if those fields
  are missing, the error handler dies with States.Runtime exactly when it's
  supposed to save the run.
  3. ManifestFallback likely breaks downstream: it substitutes raw segments
  lacking cropPlanUrl/focusTarget, which BuildSplitPayload requires вҖ” unless
  CalculateSegments emits those fields (even as nulls).
  4. Every $$.Execution.Input.* field is hard-required (bgmVolume, projectTitle,
  aspect ratios, etc.) вҖ” omitting any from the start input is an uncatchable
  failure. Only enableAnalysis is guarded with IsPresent.

  Smaller notes: UpdateStatusFailed has no Catch (a Convex outage masks the real
  error), and AssembleFinalOutput deliberately uses the pre-enrichment
  segmentsResult.manifestUrl rather than the enriched manifest вҖ” confirm that's
  intended.

  probes, computes segments (I ported the exact marker/chunk/duration logic from
  E2E-calculate-segments), then per short: trims with stream copy, extracts
  audio, gets an SRT from your existing Flux-TTS-S2T endpoint (transcribe mode,
  word timestamps), builds the styled karaoke ASS, does one NVENC encode that
  combines 9:16 conversion + 1080Г—1920 scaling + caption burn, adds the
  title/end slides, mixes BGM (from a bgm_url or the bgm mode on the same Flux
  endpoint), and uploads to R2 under your storystudio/{video,voice,txt,bgm}/
  convention. Completion comes back via webhook to Convex.

  What the Lambda code analysis revealed (this is why the rewrite pays off):
  - E2E-split-video re-downloads the entire long-form video once per segment,
  and validate-and-probe copies it to S3 first вҖ” a 4-short job transfers the
  full source 5 times. The worker downloads it once.
  - Each short is re-encoded up to 4 times (convert вҶ’ upscale вҶ’ caption burn вҶ’
  slide concat). The worker does one video encode per short; the upscale Lambda
  disappears entirely because the convert step encodes directly at 1080Г—1920.
  - Captions improve for free: the SFN slices a global SRT and clamps cues at
  segment boundaries (mid-sentence cuts), and its karaoke effect fakes word
  timing by even-splitting each cue. Per-short transcription gives real word
  timestamps native to each clip.

  bgm_volume 0.18.
